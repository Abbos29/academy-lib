Web accessibility

Accessibility

Что означает термин Accessibility?

В контексте веба это означает доступность сайта для людей с ограниченными возможностями.

Цель accessibility - сделать сайт доступным для людей с такими расстройствами как когнитивные, нейрологические, физические, расстройствами речи, для слабовидящих или незрячих.

Большинство веб-разработчиков часто пренебрегают принципами accessibility, но это все равно является достаточно важной частью веб-разработки, и эту тему нужно знать и понимать.

Web accessibility основывается на так называемых принципах POUR:

  • Perceivable. Воспринимаемость. Контент в сети должен быть воспринимаемым. Это означает, что сайт должен быть доступен для органов чувств (зрение, осязание и слух) либо через браузер, либо с помощью вспомогательных технологий, таких как screen reader, увеличители экрана и т.д.

    Примеры. Статья представлена ​​в виде отсканированного PDF документа. Если пользователь не может открыть PDF, как он ее иначе может прочитать? Контактная информация сайта отображается в футере. Если контраст между фоном футера и текстом слишком небольшой, как может пользователь с плохим зрением или дальтоник связаться с владельцем сайта?

  • Operable. Управляемость. Содержимое сайта должно быть управляемым. Это означает, что пользователи могут взаимодействовать со всеми элементами управления и интерактивными элементами, используя мышь, клавиатуру или вспомогательное устройство.

    Примеры. Меню навигации имеет еще суб-меню, которое появляется, когда пользователь наводит на него курсор мыши. Если оно не раскрывается при получении фокуса, как пользователь клавиатуры может получить доступ к нему?

  • Understandable. Понятность. Содержание в Интернете должно быть четким, кратким и простым для понимания. Постарайтесь как можно меньше допускать орфографических ошибок и применения сложных формулировок и конструкций.

    Примеры. Веб-сайт использует многочисленные сокращения, аббревиатуры и жаргон. Если они нигде не определены и не раскрыто их значение, как пользователи с ограниченными возможностями (или другие люди "не в теме") могут их понять?

    Форма регистрации содержит обязательные поля - электронная почта и номер телефона. Если форма не сообщает пользователю, в чем именно произошла ошибка валидации при вводе, как пользователь может понять, почему форма не отправляется?

  • Robust. Надежность. Контент должен быть надежным в такой степени, которая требуется для его интерпретации широким кругом различных пользовательских приложений, включая ассистивные технологии. Это означает, что вы разрабатываете, учитывая особенность инструментов, для которых вы его создаете. Работает ли ваш сайт одинаково на всех платформах, браузерах операционных системах, адаптивен для разных размеров устройств и т.д.?

    Примеры. Сайт требует определенной версии браузера, для использования его функций. Если пользователь не использует эту версию браузера, может ли он пользоваться сайтом?

Web accessibility - достаточно обширная тема. Но для того, чтобы уже сегодня начать более ответственно вести разработку, достаточно ввести в практику несколько простых вещей, и помнить о них, при разработке:

  1. Используйте атрибут alt в теге img.

    Много кто игнорирует атрибут alt, который согласно различным спецификациям является обязательным.

    Допустим, у нас есть картинка, которую нужно разместить на сайте. Цель атрибута alt - установка альтернативного текста для изображений, в случае если изображение по какой-то причине недоступно (например, не загрузилось).

    Напишем код:

    <img src="./car.png" alt="Red Ford car">

    Вот как будет выглядеть данный код, если изображение по какой-то причине не загрузится:

    Разница между наличием атрибута alt на всех изображениях и его отсутствием заключается в том, что он обеспечивает бОльшую доступность вашего веб-сайта, поскольку теперь screen reader-ы могут объявлять теги alt для людей с нарушениями зрения и когнитивных функций. Одно из практических правил заключается в том, чтобы альтернативное описание описывало более-менее детально, что изображено на картинке. В приведенном выше примере написано Red Ford car, а не просто car.

  2. Пишите семантичный HTML

    Это еще одно важное правило, которое множество разработчиков игнорирует. Они используют div для верстки всего и забывают о соответствующей разметке для заголовков, списков, таблиц и т.д. HTML5 предоставляет много семантичных тегов, таких как <nav>, <section>, <header>, <footer>, <aside> чтобы лучше структурировать ваш контент. Вот пример хорошо структурированной страницы с использованием семантического HTML:

    <body>
  <header>
    <h1>DANIT Education</h1>
  </header>
  <main>
    <article>
      <h2>Article about Web accessibility</h2>
      <p>Please, use semantic HTML for your layouts</p>
    </article>
  </main>
  <footer>
    <p>© All rights reserved</p>
  </footer>
</body>

  3. Определяйте язык веб-страницы. Не забывайте указывать основной язык каждой страницы, используя атрибут lang в корневом теге HTML, например, <html lang="en">. Используйте атрибут lang для определенных элементов, если язык текста в элементе отличается от языка остальной части страницы. Подробнее про атрибут lang можно почитать здесь.

  4. Используйте заголовки (h1-h6) только для разделения содержимого страницы. Не используйте их для выделения текста (например, сделать какой-то текст жирным или больше по размеру). Для этого необходимо использовать исключительно CSS.

  5. При добавлении изображения определитесь, предназначено ли оно только для декоративных целей, или оно очень важно для контекста. Если цель является чисто декоративной, используйте свойство background-image в CSS, а если очень важен контекст изображения - тег <img>.

  6. Избегайте использования таблиц для верстки. Многие верстальщики до сих пор используют таблицы для верстки, в то время как другие считают это плохой практикой. С точки зрения accessibility верстка таблицами - плохой подход. Таблицы создают много проблем для устройств, которые анализируют и читают страницу. Старайтесь использовать таблицы только для табличных данных.

  7. Используйте тег <label> для элементов формы (инпуты, чекбоксы, селекты и т.д). Когда элементы формы имеют подобные лейблы, это делает их более удобными для скрин ридеров, и для пользователей с ограниченными возможностями так проще в них ориентироваться.

  8. Показывайте сообщение пользователям, которые отключают JavaScript. Если для выполнения определенного действия или просмотра какой-либо информации требуется JavaScript, обязательно укажите это в сообщении. Если есть возможность сделать сайт так, чтобы пользователи могли выполнять основные действия без включения Javascript, делайте это.

  9. Проверяйте код валидаторами. Наиболее известный и понятный валидатор - W3C. Он покажет ошибки по семантике и выдаст рекомендации как их исправить.

Тема Accessibility достаточно обширная и объемная. Создание a11y сайтов - это задача не только верстальщика. Это целый подход, который включает в себя продуктовую разработку, дизайн и, собственно, само создание сайтов. Некоторые компании не применяют эти подходы. В таких же странах, как, например, США, на законодательном уровне прописано, что сайт обязательно должен соответствовать этим принципам.
